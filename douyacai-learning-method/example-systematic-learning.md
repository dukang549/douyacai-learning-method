# Example: Systematic Learning

## User request

```text
我想系统学习 API。不要只给我定义，我想建立一个完整框架。
```

## Expected mode

系统学习

## Why this mode

用户明确说了“系统学习”和“建立完整框架”，这时应该触发路线图，而不是只给一次性解释。

## Good response shape

```md
我们先不急着把所有细节一次讲完，先把 API 这个主题拆成学习路径。

📍 学习路线图：API

模块1：API 是什么
理解 API 的本质、作用和常见使用场景。 ⬜ 未开始

模块2：请求与响应
理解客户端、服务端、请求、响应这些最基础的交互关系。 ⬜ 未开始

模块3：常见 API 形式
区分 REST、GraphQL、RPC 等常见风格。 ⬜ 未开始

模块4：认证与权限
理解 API key、Token、OAuth 这些访问控制方式。 ⬜ 未开始

模块5：如何实际调用 API
从文档、参数、返回结果到错误处理，建立真实使用能力。 ⬜ 未开始

如果这个路线图符合你的预期，我们就从模块1开始。第一轮我先只讲最关键的 3 个点，然后让你用自己的话复述。
```

## What should happen next

- 用户确认路线图后，再开始模块 1
- 第一轮知识点不要过多
- 讲完后必须发起复述

## What should not happen

- 不要一上来把 5 个模块全部讲透
- 不要跳过复述
- 不要在第一轮就生成 Anki

## Design note

这个案例用来说明：当用户要的是完整框架时，路线图是必要的，因为它解决的是“以后怎么学”，而不只是“这次先讲什么”。
